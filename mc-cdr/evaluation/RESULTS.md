## Known Limitations
- Dataset is entirely synthetic — real-world performance will differ
- Contamination parameter tuned on synthetic data with clean separation; expect lower precision in production
- Azure coverage limited to 4 operation types in current rule set
- Latency benchmarks measured in-process; real pipeline latency includes CloudTrail delivery delay of 2–5 minutes

## Coverage Gaps
The following rules had zero detections in the benchmark run because the synthetic
dataset does not include scenarios that trigger them:
- `azure_diagnostic_deleted` — no Microsoft.Insights/diagnosticSettings/delete events in dataset
- `azure_guest_user_added` — no Azure guest user creation events in dataset
- `azure_role_assignment` — no Azure roleAssignments events in dataset
- `cloudtrail_tampering` — no StopLogging detections in the benchmark ruleset path used for per-scenario counting
- `cross_account_role_assumption` — no AssumeRole events in dataset
- `keyvault_secrets_read` — no KeyVault read events in dataset
- `mass_access_key_list` — no ListAccessKeys events in dataset
- `root_access_key_created` — no CreateAccessKey events in dataset
- `s3_bucket_made_public` — no PutBucketAcl events in dataset
- `security_group_open` — no AuthorizeSecurityGroupIngress events in dataset

These rules are valid and tested in unit tests. Closing these gaps requires
adding corresponding attack scenarios to dataset_generator.py.
